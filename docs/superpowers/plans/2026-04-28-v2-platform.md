# SOC Triage AI → SOC Triage Copilot: Full-Stack Platform Migration

## Context

Today the project is a CodePath final that's been productized into a single-process Streamlit MVP: 7/7 reliability harness, 34 unit tests, static 11-doc threat-intel corpus, Anthropic + sentence-transformers + numpy in one Python process. It looks and behaves like a polished prototype, but it reads as "a Python class project with a UI" on a resume.

The goal is to turn it into **SOC Triage Copilot — Full-Stack AI Incident Response Platform** that demonstrates *five* engineering surfaces in one repo:

1. Applied AI / RAG with guardrails and reproducibility
2. Backend API design (FastAPI + Pydantic + OpenAPI)
3. Frontend product delivery (Next.js + TypeScript)
4. Data persistence + schema + ingestion pipelines (Postgres + pgvector)
5. Async orchestration + DevOps + observability (Celery + Redis + Docker + GitHub Actions + Terraform + OpenTelemetry)

The current AI engine (`triage.py`, `extractors.py`, `case_package.py`, `rag/`) is *kept and vendored* into a worker. None of the AI-domain code is rewritten. The platform code is built *around* it.

**Outcome:** a portfolio-grade repo whose top-level README justifiably opens with "Built a full-stack SOC triage platform using Python, FastAPI, Next.js, TypeScript, PostgreSQL, Redis, and Docker..."

---

## Architectural Spine: Three Boundaries That Don't Move

**B1. The v1 engine is a vendored Python package, not a microservice.**
Existing files move verbatim into `services/triage-worker/triage_engine/`. The Celery worker imports it. FastAPI never imports it. Frozen prompt, frozen retriever, frozen harness stay exactly as they are — only the *caller* changes.

**B2. FastAPI handlers are thin: validate → enqueue → read DB → return. No business logic.**
The only place that runs LLM calls or embeddings is the worker. API horizontally scales, `GET` p95 stays under 50ms, and the harness can target the worker library directly without HTTP.

**B3. The case package JSONB is the source of truth for the *triage moment*. `analyst_overrides` is an append-only log of post-hoc edits.**
Reads materialize the "current view" by applying overrides over the immutable envelope. Resolves the JSONB-vs-relational tension cleanly: never lose the original LLM output for audit/eval, but support fast filterable queries.

### Phase 1 — FastAPI + Next.js skeleton, synchronous triage (1 week) ← detailed

This is the next phase to execute. The rest are sketched.

#### 1.1 Postgres schema (Alembic-managed)

Nine tables, all `tenant_id UUID NOT NULL` reserved for v3 multi-tenancy. **`pgvector` extension enabled day one; `halfvec(384)` for embeddings; HNSW index.** Numpy stays in the worker for v1 corpus until Phase 4 makes pgvector retrieval the primary path.

| Table | Purpose | Notable columns |
|---|---|---|
| `alerts` | Raw inbound | `raw_text`, `source`, `external_id`, `received_at` |
| `observables` | Per-alert IOCs | `kind`, `value` (indexed for pivots) |
| `cases` | One per triaged alert | `envelope JSONB` (immutable), denormalized `severity`/`escalate`/`retrieval_score`, FK to `corpus_versions` |
| `evidence_links` | Case ↔ chunk citations | `chunk_id`, `score`, `cited` |
| `analyst_overrides` | Append-only edits | `field`, `old_value JSONB`, `new_value JSONB`, `rationale` |
| `intel_chunks` | Corpus, embedded | `embedding halfvec(384)`, `corpus_version_id`, `source_type` |
| `corpus_versions` | Provenance | `embedding_model`, `chunk_count`, `is_active` (unique-where TRUE) |
| `eval_runs` | Harness history | `metrics JSONB`, `per_case JSONB`, `git_sha` |
| `audit_logs` | Every API mutation | `actor_id`, `action`, `resource`, `payload JSONB` |
| `users`, `api_keys` | Auth | bcrypt + argon2id hashes |

Indexes: `cases (severity, created_at DESC)`, `cases USING GIN (envelope jsonb_path_ops)`, `intel_chunks USING hnsw (embedding halfvec_cosine_ops)`, `evidence_links (case_id)`, `audit_logs (actor_id, created_at DESC)`.

#### 1.2 FastAPI module layout

```
apps/api/src/soc_api/

  config.py                # pydantic-settings, env-driven
  db.py                    # async SQLAlchemy engine + session dep
  deps.py                  # auth deps (current_user, current_api_key)
  routers/
    alerts.py              # POST /alerts          (api-key auth)
    triage.py              # POST /triage/jobs     (sync in P1, queued in P3)
    cases.py               # GET /cases/{id}, POST /cases/{id}/override (P2)
    eval.py                # GET /eval/latest
    retrieval.py           # GET /retrieval/debug?query=
    corpus.py              # GET /corpus/versions
    health.py              # GET /healthz, /readyz
  services/
    alert_service.py
    case_service.py
    override_service.py
  models/orm.py            # SQLAlchemy declarative
  middleware/
    request_id.py
    audit.py
  migrations/versions/     # Alembic
```

**P1 implementation note:** `POST /triage/jobs` runs **synchronously in-process** by importing the engine. This gets a working end-to-end demo in week 1 without standing up Celery. Phase 3 swaps the implementation; the API contract stays the same.

#### 1.3 Contracts package (Pydantic → JSON Schema → TS)

```
packages/contracts/src/soc_contracts/
  alerts.py                # AlertIngestRequest, AlertIngestResponse
  triage.py                # TriageJobRequest, TriageJobResponse
  cases.py                 # CaseEnvelope (mirrors case_package output), OverrideRequest
  eval.py, retrieval.py, corpus.py
```

Generation pipeline: `datamodel-code-generator` (Pydantic → JSON Schema), then `json-schema-to-typescript` (JSON Schema → TS). CI step `make contracts` fails on diff — forces intentional contract bumps. Published as private `@soc/contracts` npm consumed by `apps/web` and as `soc-contracts` Python package consumed by `apps/api` + `services/triage-worker`.

**Why not `openapi-typescript`:** ties TS types to FastAPI's OpenAPI doc, which couples the worker to the API's route layer. Pydantic-first contracts keep the worker decoupled.

#### 1.4 Next.js console (P1 scope)

```
apps/web/src/
  app/
    page.tsx                # dashboard: recent cases table
    submit/page.tsx         # alert submission form
    cases/[id]/page.tsx     # case detail: triage card + observables + evidence
    eval/page.tsx           # latest harness run summary
  components/
    CaseCard, ObservablesPanel, EvidenceList, SeverityBadge, MitreChip
  lib/api.ts                # typed fetch wrapper, imports @soc/contracts
  lib/auth.ts               # NextAuth credentials provider
```

Four pages in P1. Override UI ships in P2 (avoids a half-done feature spanning phases).

#### 1.5 Auth posture

- **Web:** NextAuth credentials → `users` table; one bootstrapped admin via env var; bcrypt hashes; session cookies.
- **Programmatic:** `Authorization: Bearer <key>` → `api_keys` table for `POST /alerts`; argon2id hashes; fixed scopes.
- **Deferred to v3:** real multi-tenancy, SSO/OIDC, RBAC beyond admin/analyst, per-key custom scopes, per-tenant rate limiting.

Honest, demoable, no week-long OAuth detour.

#### 1.6 Migration tooling: **Alembic**

SQLAlchemy is already in the API; Alembic is its native migration tool; autogenerate covers 90% of changes; `op.execute()` handles pgvector index DDL. Atlas is more powerful but adds a Go binary + DSL — not worth the cognitive load for a solo project.

#### 1.7 P1 verification checkpoints

2. `POST /alerts` with the gibberish T6 case → 200, alert row created.
3. `POST /triage/jobs` with the LSASS T3 alert → returns case envelope, persisted.
4. `GET /cases/{id}` → returns the same envelope.
5. Browse `localhost:3000/cases/{id}` → renders triage card with MITRE chips.
6. **Harness 7/7** by running directly against `triage_engine` (no HTTP) — proves the engine wasn't perturbed by the move.

### Phase 2 — Override UI + persistence loop (3–4 days)

**Goal:** analyst feedback loop without touching the LLM path.

- `apps/api/src/soc_api/routers/cases.py` → `POST /cases/{id}/override`, validates field whitelist, appends to `analyst_overrides`, writes `audit_logs`.
- `case_service.py:get_materialized_case(case_id)` applies overrides over envelope (latest override per field wins).
- `apps/web/src/app/cases/[id]/EditPanel.tsx` — inline edit for severity, escalate, MITRE; rationale required.
- `apps/web/src/app/cases/[id]/HistoryPanel.tsx` — override timeline.

**Key decision:** override application is read-side, not write-side. `cases.envelope` JSONB never changes; materialized view computed per request. Cache later if needed.

**Demo:** edit a case, reload, see the new value + a history entry. Audit log query returns the action.

### Phase 3 — Celery + Redis, async triage (1 week)

API stops blocking on LLM calls.

```
services/triage-worker/src/soc_worker/
  celery_app.py             # Celery() with Redis broker
  tasks/
    triage_task.py          # @celery_app.task triage_alert(alert_id)
    ingest_task.py
  engine_loader.py          # singleton SOCTriage instance, lazy-loaded
  triage_engine/            # vendored from Phase 0
```

**Critical decision: workers preload the model at process start.**
sentence-transformers takes ~30s to load. Per-task load is unworkable. Process-level singleton initialized in Celery's `worker_process_init` signal. Each worker process holds ~90MB. Run 2–4 workers initially; scale by adding processes (embedding is CPU-bound, model not thread-safe under load).

**No separate model service.** Adds RPC hop, doesn't help until ≥10 worker processes. Defer until that's a real problem.

API changes: `POST /triage/jobs` enqueues, returns `{job_id, status: "queued"}`. New `GET /triage/jobs/{job_id}` for polling. Web uses SWR with 1s poll until `completed`.

**Demo:** submit 5 alerts in parallel from the UI → all complete within ~5s ceiling. Harness 7/7 (run via direct Python import of the engine, *not* through Celery — the harness tests the engine, not the queue).

### Phase 4 — MITRE ATT&CK + CISA KEV ingestion + corpus versioning (1 week)

**Goal:** static markdown corpus → live, scheduled, versioned intel.

```
jobs/intel-sync/src/intel_sync/
  sources/mitre_attack.py     # stix2 lib against MITRE TAXII
  sources/cisa_kev.py         # plain JSON fetch
  chunkers/{mitre,kev}_chunker.py
  embedder.py                 # batch embed via sentence-transformers
  versioner.py                # creates corpus_versions row, atomically swaps is_active
  cli.py                      # intel-sync run --sources mitre,kev
```

**Decisions:**
- **STIX parsing:** use the `stix2` library. MITRE bundle has nested relationships; rolling our own JSON path parsing is a worse trade than the lib's ergonomics tax.
- **CISA KEV:** daily ingest. Cron via GitHub Actions or Celery beat. KEV publishes 1–3 entries/day; webhook isn't offered.
- **Corpus versioning is atomic at activation, not at write.** Ingest into `intel_chunks` with a *new* `corpus_version_id`, run the harness against it, *then* flip `is_active`. Worker reloads index on Redis pub/sub signal or next process recycle.
- **Retrieval source switch:** worker now reads from pgvector (`PgvectorRetriever` adapter implementing the same `.retrieve(query, top_k)` contract). The frozen `ThreatIntelRetriever` interface is preserved by adapter, so the harness doesn't notice.

**Demo:** corpus grows 109 → 800+ chunks. `/corpus` page shows version diff. Harness 7/7 against the live corpus (or fails informatively).

### Phase 5 — Production plumbing: Docker, GitHub Actions, Terraform, OpenTelemetry (4–5 days)

- `.github/workflows/ci.yml` — unit tests on every push; **harness on PRs labeled `run-harness` and on a nightly cron** (cost guard: 7 Anthropic calls per run × N PRs adds up)

- OpenTelemetry: trace context propagated through Celery headers (API → worker → DB); structured JSON logs; export to free tier (Honeycomb sandbox or self-hosted Jaeger in compose).

**Open decisions** flagged for Solomon:

Maintenance cost near zero — engine interface is frozen. Delete only if it requires >30 min of upkeep in any quarter.

## Locked vs Deferred

**Locked in this v2 migration:**
Monorepo layout · FastAPI + Next.js · Postgres + pgvector + Redis · Alembic · Celery (no separate model service) · Pydantic→JSON Schema→TS contract pipeline · sentence-transformers in worker · single-tenant + API key + admin/analyst auth · Streamlit kept as internal dev console · halfvec(384) + HNSW · JSONB envelope as triage source-of-truth + append-only overrides · harness 7/7 as phase-exit gate · harness on PR-label + nightly only.

**Deferred to v3+:**
Real multi-tenancy (column reserved, value hardcoded) · SSO/OIDC · fine-grained RBAC · model service · Redis Streams · integrations beyond Elastic · model A/B harness comparison · prompt versioning UI · auto-retraining loops · customer-managed corpora.

**Hard "no":**
Embedding model swap inside v2 (would break frozen retriever score contract; treat as a separate experiment) · multi-region (RDS single-AZ fine for portfolio) · streaming LLM responses (case envelope is a discrete artifact; streaming adds complexity for no analyst value).
